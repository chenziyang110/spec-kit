# Checklist 详解：形象表达 + 硬核拆解

本文讲 Spec 方法论中的 `checklist` 这一步。

目标：

- 理解它为什么不是“测试代码”
- 理解它为什么更像“需求质量检查”
- 理解它在整个流程中扮演的门禁角色

本文全部使用普通 Markdown 与 ASCII 图。

---

# 一、`checklist` 是什么

一句话：

**`checklist` 不是在验证实现，而是在验证需求、规格和质量要求是否写得足够清楚。**

它的本质更接近：

```text
requirements QA
而不是 runtime QA
```

---

# 二、先看图

```text
spec / plan / tasks
       ↓
+------------------+
|    checklist     |
|  生成质量检查单   |
+------------------+
       ↓
作为 implement 前的门禁之一
```

核心意思：

`checklist` 不是结果层，
它是门禁层。

---

# 三、形象表达

## 1. 它像什么

### 比喻 1：像“英文需求的单元测试”

这其实就是模板里对它的定位。

它不是测代码跑没跑，而是测：

- 需求有没有写全
- 有没有写清
- 有没有前后冲突
- 有没有缺边界情况

所以它更像：

**对文档质量做单元测试。**

### 比喻 2：像验收前的检查表

在真正开工前，先拿一张表检查：

- 安全要求写了吗
- UX 状态写了吗
- 非功能要求是不是量化了
- 边界情况是不是漏了

它是防止“带着模糊文档进入实现”。

---

# 四、硬核拆解

## 1. 输入是什么

它通常会读取：

- `spec.md`
- `plan.md`
- `tasks.md`

同时再结合用户指定的关注维度，例如：

- UX
- security
- performance
- API

---

## 2. 输出是什么

输出通常位于：

```text
specs/<feature>/checklists/*.md
```

例如：

- `requirements.md`
- `ux.md`
- `security.md`
- `performance.md`

---

## 3. 它的关键机制

它检查的不是“实现对不对”，而是“需求写得好不好”。

典型检查维度：

- Completeness
- Clarity
- Consistency
- Measurability
- Coverage

这就是为什么它能在实现前就起作用。

---

## 4. 最后两句总结

### 形象版

**`checklist` 像开工前的需求质量检查表，不是测程序跑没跑，而是测文档能不能支撑后续工作。**

### 硬核版

**`checklist` 是门禁层：它围绕 spec/plan/tasks 生成针对需求质量的检查项，用来阻止模糊或不完整的输入直接流入实现阶段。**

# Constitution 详解：形象表达 + 硬核拆解

本文只讲 Spec 方法论中的 `constitution` 这一步，并用两种方式解释：

- 形象表达：帮助快速抓住感觉
- 硬核拆解：帮助准确理解结构与作用机制

同时，本文全部使用普通 Markdown 和 ASCII 图，不依赖 Mermaid 渲染。

---

# 一、`constitution` 到底是什么

一句话：

**它不是在定义“这个功能做什么”，而是在定义“这个项目以后做事按什么最高规则来判断”。**

你可以把它理解成：

- 项目的宪法
- 项目的裁判手册
- 后续所有步骤的总约束

---

# 二、先看图

## 1. 它在整个流程里的位置

```text
用户/团队原则
     ↓
+------------------+
|  constitution    |
|  定规则与门禁     |
+------------------+
     ↓
+------------------+
|   specify        |
|  定义做什么       |
+------------------+
     ↓
+------------------+
|   clarify        |
|  消除歧义         |
+------------------+
     ↓
+------------------+
|     plan         |
|  设计怎么做       |
+------------------+
     ↓
+------------------+
|     tasks        |
|  拆成可执行任务   |
+------------------+
     ↓
+------------------+
|   implement      |
|  真正开始执行     |
+------------------+
```

核心意思：

**`constitution` 不生产功能，它生产“规则”。**
后面的每一步都要在这个规则里运行。

---

## 2. 它在系统里的角色

```text
constitution = 元规则层
specify      = 目标层
clarify      = 消歧层
plan         = 方案层
tasks        = 拆解层
implement    = 执行层
analyze      = 校验层
```

所以它最像：

```text
操作系统 / 宪法 / 裁判规则
而不是一个业务功能
```

# 三、形象表达

## 1. `constitution` 像什么

### 比喻 1：它是“项目宪法”

这不是项目说明书，也不是产品需求文档。

它更像国家宪法：

- 不规定今天修哪条路
- 不规定明天买什么东西
- 它规定的是：
  - 哪些原则高于一切
  - 冲突时按什么裁决
  - 什么行为是非法的
  - 修改规则要走什么程序

所以：

- `specify` 像立法提案
- `plan` 像实施方案
- `tasks` 像施工清单
- `implement` 像开工
- `constitution` 像“这个国家怎么运作”

### 比喻 2：它是“裁判手册”

项目后面每一步都会出现分歧：

- 要快还是要稳
- 要简单还是要可扩展
- 要先交付还是先测试
- 要不要接受模糊需求
- 出现例外时谁能破例

如果没有 `constitution`，每个人都在临场当裁判。
有了它，相当于你先把裁判标准写死。

### 比喻 3：它是“主操作系统，不是应用”

后面每个命令都像 App：

- `specify` 是需求 App
- `plan` 是设计 App
- `tasks` 是拆解 App
- `implement` 是执行 App

而 `constitution` 像底层 OS。
它不直接干业务，但所有业务都运行在它上面。

---

## 2. 它解决的根本问题

它解决的是：

**团队或 AI 没有统一的“判断标准”。**

没有它时，常见情况是：

- 需求写的人觉得“差不多就行”
- 设计的人觉得“要先做扩展性”
- 拆任务的人没加测试
- 实现的人为了省事跳过约束
- 最后大家都觉得自己没错

问题不在能力，而在于：

**没有共同的最高规则。**

`constitution` 就是先把这个最高规则钉住。

---

## 3. 最简单的一句话理解

如果你只记一句：

**`constitution` 不是告诉你“做什么”，而是告诉你“以后怎么判断对不对”。**

# 四、硬核拆解

## 1. 输入是什么

`constitution` 这一步的输入通常来自 3 类：

### 1. 用户显式输入

例如：

- 我们优先代码可维护性
- 必须测试优先
- 不能引入新依赖
- 要适合非技术用户

### 2. 仓库已有上下文

例如：

- README
- docs
- 现有项目结构
- 历史宪章版本

### 3. 模板结构

核心模板在：

[constitution-template.md](/F:/github/spec-kit/templates/constitution-template.md)

模板里有占位符，比如：

- `[PROJECT_NAME]`
- `[PRINCIPLE_1_NAME]`
- `[PRINCIPLE_1_DESCRIPTION]`
- `[GOVERNANCE_RULES]`

这意味着它不是从零凭空写，而是：
**在一个规范骨架上做具体化。**

---

## 2. 输出是什么

它的输出不是“建议”，而是一个正式约束文件：

```text
.specify/memory/constitution.md
```

这份文件通常包含：

### A. 核心原则

例如：

- 简单优先
- 测试优先
- 可验证优先
- 安全不可跳过

### B. 附加约束

例如：

- 合规要求
- 性能门槛
- 依赖限制
- 技术限制

### C. 治理规则

例如：

- 原则如何修改
- 版本号如何升级
- 例外如何审批
- 谁负责复核

### D. 版本信息

例如：

- Version
- Ratified
- Last Amended

所以它是一个“带版本的规则文件”。

---

## 3. 它的内部执行流程

命令流程定义在：

[constitution.md](/F:/github/spec-kit/templates/commands/constitution.md)

核心流程如下：

```text
[开始]
   ↓
读取现有 constitution
   ↓
如果不存在
   → 用 constitution-template 初始化
   ↓
扫描占位符
   ↓
收集规则来源
   ├─ 用户输入
   ├─ 仓库上下文
   └─ 历史内容
   ↓
生成原则与治理文本
   ↓
判断版本升级级别
   ├─ MAJOR
   ├─ MINOR
   └─ PATCH
   ↓
检查联动影响
   ├─ spec-template
   ├─ plan-template
   ├─ tasks-template
   └─ commands/*
   ↓
生成 Sync Impact Report
   ↓
最终校验
   ├─ 占位符是否残留
   ├─ 日期是否正确
   ├─ 版本是否一致
   └─ 原则是否清晰可执行
   ↓
写回 .specify/memory/constitution.md
   ↓
[结束]
```

---

## 4. 每一步具体在做什么

### 第 1 步：读取现有宪章

目标文件：

```text
.specify/memory/constitution.md
```

如果没有，就从模板创建。

这一步不是为了写内容，而是拿到一个“规则骨架”。

### 第 2 步：扫描占位符

系统会识别模板中的槽位，例如：

```text
[PROJECT_NAME]
[PRINCIPLE_1_NAME]
[SECTION_2_NAME]
[GOVERNANCE_RULES]
```

本质是在确定：

**有哪些结构位置必须被具体化。**

### 第 3 步：收集规则内容

内容来源优先级通常是：

```text
用户明确要求
   > 仓库已有上下文
   > 旧版规则
   > 必要时 TODO
```

这一层做的是：

- 提炼真正关心的判断标准
- 避免空泛口号
- 把隐含偏好显式化

### 第 4 步：决定版本升级

它要求规则变更也做版本管理：

- `MAJOR`
  - 删除原则
  - 重定义原则
  - 改变原有治理逻辑
- `MINOR`
  - 新增原则
  - 新增章节
  - 新增门禁
- `PATCH`
  - 改措辞
  - 补澄清
  - 修小问题

这说明它把“治理变化”当成正式变更，而不是随手改文案。

### 第 5 步：生成正文

这一层真正写的是：

- 原则
- 限制
- 治理方式

要求通常是：

- 清楚
- 尽量可验证
- 不模糊
- 不写成鸡汤
- 不写成功能需求

### 第 6 步：检查联动影响

它不会只改宪章本身，还会看它是否影响：

- `spec-template.md`
- `plan-template.md`
- `tasks-template.md`
- `templates/commands/*.md`

原因是：

如果你在宪章里写了规则，但后面的模板和流程完全不体现，那这条规则就是空的。

所以这一步本质上是在：

**把规则传播到工作流层。**

### 第 7 步：生成影响报告

会记录：

- 旧版本 -> 新版本
- 改了哪些原则
- 增了哪些章节
- 删了哪些章节
- 哪些模板已同步
- 哪些还待处理

作用是：

**让规则变更的影响范围可见。**

### 第 8 步：最终校验

主要检查：

- 是否还有未解释占位符
- 日期格式对不对
- 版本号是否匹配
- 原则有没有写得太虚
- 结构有没有损坏

相当于：

**先保证规则本身稳定，再让它去约束下游。**

---

## 5. 它怎么影响后面的步骤

```text
constitution
   │
   ├── 约束 specify
   │     - 需求怎么写
   │     - 成功标准怎么定义
   │     - 是否允许模糊词
   │
   ├── 约束 plan
   │     - 技术方案要不要过门禁
   │     - 是否必须考虑风险/性能/安全
   │
   ├── 约束 tasks
   │     - 是否必须有测试任务
   │     - 是否必须有文档/安全/验证任务
   │
   ├── 约束 implement
   │     - 哪些事不能跳过
   │     - 哪些质量标准必须满足
   │
   └── 约束 analyze
         - 宪章冲突直接视为高优先级问题
```

其中影响最大的是 `plan`。

因为 `plan` 模板里会有：

```text
Constitution Check
```

也就是说方案不是自由发挥，而是：

**先问：这个方案符不符合项目最高规则？**

---

## 6. 什么样的 constitution 才算好

好的 `constitution` 通常具备这些特征：

### 特征 1：短

不求长，求高密度。

### 特征 2：硬

要能裁决，不要空话。

### 特征 3：清楚

能看出行为约束。

### 特征 4：可执行

后面的人知道怎么照着做。

### 特征 5：可验证

别人能判断你到底有没有遵守。

---

## 7. 差和好的对比

### 差的写法

- 保持高质量
- 注重用户体验
- 关注性能
- 尽量简单

这些都太空。

### 好的写法

- 所有新增行为必须有对应验证方式
- 不允许使用“快”“稳定”“安全”这类不可衡量词汇作为验收标准
- 高风险变更必须提供回滚路径
- 未经明确批准，不得引入新依赖
- 测试可以后补，除非涉及高风险路径

这种写法才能真正指导后续流程。

---

## 8. 最值得写什么

最值得写的不是“所有想到的东西”，而是：

**以后会反复影响判断的东西。**

建议优先写 4 类：

### 1. 价值排序

例如：

- 可维护性高于交付速度
- 准确性高于自动化程度
- 简单优先于过早抽象

### 2. 质量门禁

例如：

- 所有新增行为必须可验证
- 高风险改动必须有测试
- 非功能要求必须量化
- 重大变更必须可回滚

### 3. 外部约束

例如：

- 不加新依赖
- 必须离线可用
- 必须兼容旧系统
- 必须可审计

### 4. 治理规则

例如：

- 例外必须显式记录
- 原则修改必须更新版本
- 高影响变更必须复审
- 模板与流程需要跟随规则同步

---

## 9. 最常见误区

### 误区 1：写成愿景口号

`constitution` 不是：

- 我们追求卓越
- 我们以用户为中心

这些都不能裁决冲突。

### 误区 2：写成功能需求

例如：

- 系统支持上传图片
- 系统支持搜索

这属于 `specify`，不是 `constitution`。

### 误区 3：写成实现手册

例如：

- 必须用某框架
- 必须按某种目录结构
- 接口必须怎么写

这已经掉到 `plan` 层了。

### 误区 4：原则彼此冲突

例如同时写：

- 简单优先
- 一开始就为未来十倍规模设计抽象

这会让后续裁决失效。

---

## 10. 最后两句总结

### 形象版

**它是项目的宪法，不是项目的施工图。**

### 硬核版

**它把“偏好”升级成“规则”，把“经验”升级成“门禁”。**

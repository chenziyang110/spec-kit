# Taskstoissues 详解：形象表达 + 硬核拆解

本文讲 Spec 方法论中的 `taskstoissues` 这一步。

这是一个可选步骤，不属于主 SDD 核心链路，但对协作很有用。

---

# 一、`taskstoissues` 是什么

一句话：

**它是把 `tasks.md` 中的执行任务，映射成 GitHub Issues。**

它不负责需求、设计或实现，而是负责把“任务文档”同步到“协作系统”。

---

# 二、先看图

```text
tasks.md
   ↓
+------------------+
|  taskstoissues   |
|  任务转 issue     |
+------------------+
   ↓
GitHub Issues
```

---

# 三、形象表达

## 1. 它像什么

### 比喻 1：像“从施工计划导入工单系统”

你已经有一份内部任务计划了。

但团队协作还需要：

- issue 跟踪
- assignee
- 状态流转
- 看板管理

这一步就是把内部任务结构投射到 GitHub 的协作面板里。

---

# 四、硬核拆解

## 1. 输入是什么

主要输入：

- `tasks.md`
- 当前仓库的 git remote

## 2. 输出是什么

输出：

- 在匹配的 GitHub 仓库中创建 issue

## 3. 它的关键约束

它会先检查：

- 当前 remote 是否真的是 GitHub URL
- 创建 issue 的仓库是否与 remote 匹配

这是为了避免把任务误发到错误仓库。

---

## 4. 它为什么是可选层

因为核心 SDD 主流程只需要：

- 需求
- 方案
- 任务
- 实现

而 issue 同步属于协作系统层。

所以它是：

**协作映射层，而不是核心设计层。**

---

## 5. 最后两句总结

### 形象版

**`taskstoissues` 像把内部施工清单导入 GitHub 工单系统。**

### 硬核版

**`taskstoissues` 是协作映射层：它把 `tasks.md` 中的执行单元同步成外部 issue 系统中的跟踪对象。**

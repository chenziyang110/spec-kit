# 通用 Spec 方法论（跨领域版）

本文将 Spec Kit 背后的核心思想抽象成一套可迁移的方法论，不限定于软件开发，也适用于研究、咨询、内容生产、课程设计、运营流程设计等复杂知识工作。

目标不是复刻 `/speckit.*` 命令，而是提炼出它真正可复用的认知框架。

---

## 一、核心结论

这套方法论的本质不是“用一堆命令写代码”，而是：

**把复杂工作强制分层，让人或 AI 在每一层只做该层该做的事。**

它反对把下面这些事情混在一起：

- 定义问题
- 澄清歧义
- 设计路径
- 拆解任务
- 直接执行
- 最后才发现方向错了

它主张把它们拆成明确阶段，并让每一阶段都有单独的产物。

---

## 二、七层通用框架

任何复杂任务都可以拆成这 7 层：

1. 原则
2. 目标
3. 澄清
4. 方案
5. 拆解
6. 执行
7. 校验

你可以把它记成：

```text
原则 -> 目标 -> 澄清 -> 方案 -> 拆解 -> 执行 -> 校验
```

---

## 三、每一层分别解决什么问题

### 1. 原则

先定义不能违反的约束。

这一层回答的问题：

- 什么结果才算好结果
- 哪些红线不能碰
- 冲突时优先级怎么排
- 哪些标准高于短期效率

典型产物：

- 原则清单
- 质量标准
- 决策规则
- 禁区列表

没有这一层，后面所有判断都会漂移。

---

### 2. 目标

定义“要达成什么”，而不是“准备怎么做”。

这一层回答的问题：

- 要解决什么问题
- 为谁解决
- 为什么值得做
- 成功是什么样
- 不做什么

典型产物：

- 目标说明
- 范围定义
- 成功标准
- 场景清单

这一步要尽量避免提前掉进实现细节。

---

### 3. 澄清

专门处理模糊性。

这一层回答的问题：

- 哪些词太模糊
- 哪些边界没说清
- 哪些地方存在多种合理解释
- 哪些未决问题会显著影响后续路径

典型产物：

- 澄清问答
- 决策记录
- 假设清单
- 未决问题列表

这是最容易被跳过、但最容易减少返工的一层。

---

### 4. 方案

把目标翻译成路径。

这一层回答的问题：

- 准备用什么方法去实现目标
- 为什么选这条路
- 风险怎么处理
- 资源怎么配置
- 放弃了哪些替代方案，为什么

典型产物：

- 实施方案
- 结构设计
- 方法路径
- 风险与权衡说明

这一步才进入“怎么做”。

---

### 5. 拆解

把方案变成可执行单元。

这一层回答的问题：

- 先做什么，后做什么
- 哪些有依赖
- 哪些能并行
- 每一项交付物是什么
- 每项任务对应哪个目标

典型产物：

- 任务列表
- 依赖图
- 阶段划分
- 责任分配

拆解的重点不是“列很多事”，而是让任务真正可执行、可验证、可追踪。

---

### 6. 执行

真正开始做。

这一层的关键要求：

- 不要脱离前面产物自由发挥
- 过程中持续记录状态
- 遇到偏差时先判断是哪一层出了问题

执行不是单纯生产，而是沿着约束推进。

---

### 7. 校验

检查整套东西是否一致。

这一层回答的问题：

- 结果是否仍然符合目标
- 任务是否覆盖目标
- 方案是否违反原则
- 是否出现术语漂移、范围漂移、任务漂移
- 是否还有未处理风险

典型产物：

- 一致性报告
- 覆盖率检查
- 偏差说明
- 下一轮修正建议

这一步不是只看最终结果，还要看过程是否仍然对齐。

---

## 四、为什么它可以跨领域迁移

因为这套框架抽象的不是“软件开发步骤”，而是“复杂认知任务的分层结构”。

每一层对应的是一种不同的思维工作：

- 原则层：定义判断标准
- 目标层：定义输出方向
- 澄清层：消除歧义
- 方案层：设计路径
- 拆解层：组织行动
- 执行层：产出结果
- 校验层：修正偏差

只要你的工作满足以下特征，它就适用：

- 任务复杂
- 存在不确定性
- 需要多阶段推进
- 返工成本高
- 多人或 AI 参与协作

---

## 五、跨领域映射示例

### 1. 软件开发

- 原则 = 架构原则、质量门禁、工程规范
- 目标 = 需求规格
- 澄清 = 需求访谈、需求补充
- 方案 = 技术设计、系统方案
- 拆解 = 开发任务与里程碑
- 执行 = 编码与集成
- 校验 = 测试、代码审查、一致性分析

---

### 2. 研究项目

- 原则 = 研究边界、方法约束、伦理要求
- 目标 = 研究问题、研究假设
- 澄清 = 概念界定、变量边界澄清
- 方案 = 研究设计、方法路径
- 拆解 = 实验计划、资料收集计划
- 执行 = 调研、实验、分析
- 校验 = 结果审查、偏差分析、方法一致性检查

---

### 3. 内容生产 / 营销

- 原则 = 品牌原则、受众边界、表达禁区
- 目标 = 内容目标、受众收益、传播目的
- 澄清 = 信息重点、语气、边界确认
- 方案 = 内容结构、渠道策略、传播路径
- 拆解 = 选题、脚本、制作、发布任务
- 执行 = 创作与投放
- 校验 = 数据回看、表达一致性、目标达成分析

---

### 4. 咨询 / 解决方案设计

- 原则 = 业务约束、合规要求、交付标准
- 目标 = 客户问题定义、成功标准
- 澄清 = 访谈、范围确认、例外情况识别
- 方案 = 解决路径、阶段方案、资源设计
- 拆解 = 工作包、时间表、责任矩阵
- 执行 = 项目推进
- 校验 = 阶段回顾、偏差分析、目标对齐检查

---

### 5. 课程 / 知识产品设计

- 原则 = 教学目标、认知负荷限制、学习者约束
- 目标 = 学习结果、适用对象、边界范围
- 澄清 = 学员前置能力、场景、难度要求
- 方案 = 课程结构、模块设计、教学路径
- 拆解 = 讲义、案例、练习、录制计划
- 执行 = 制作与讲授
- 校验 = 学习反馈、完成率、目标达成情况

---

## 六、最小可用版本

如果你的任务不那么复杂，不必把 7 层全开。

最小可用版本可以只保留这 4 层：

```text
原则 -> 目标 -> 方案 -> 执行
```

当复杂度上升时，再插入：

- 澄清
- 拆解
- 校验

也就是说，它是一个可伸缩框架，不是死板流程。

---

## 七、可直接复用的通用模板

你可以把下面这份骨架直接用于任何领域：

```text
A. 原则
- 成功标准
- 约束条件
- 禁区
- 优先级规则

B. 目标
- 要解决的问题
- 服务对象
- 目标结果
- 范围边界

C. 澄清
- 关键歧义
- 待确认问题
- 核心假设
- 高风险不确定点

D. 方案
- 实施路径
- 方法选择
- 关键权衡
- 风险控制

E. 拆解
- 阶段划分
- 任务列表
- 依赖关系
- 并行机会

F. 执行
- 按阶段推进
- 记录状态
- 处理偏差

G. 校验
- 结果是否达标
- 是否覆盖目标
- 是否违反原则
- 下一步修正
```

## 八、最值得记住的一句话

这套方法论最值钱的地方是：

**不要让“定义问题、澄清问题、设计路径、执行任务”混在同一层里。**

一旦混在一起，人会乱，AI 也会乱。

把这些动作分层以后：

- 每一层只做一种认知工作
- 每一层都有自己的输出物
- 下一层只接收上一层的结果
- 出问题时也更容易定位是哪一层出了问题

这就是它能跨领域复用的根本原因。

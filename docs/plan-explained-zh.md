# Plan 详解：形象表达 + 硬核拆解

本文讲 Spec 方法论中的 `plan` 这一步。

目标：

- 理解它为什么是“方案层”
- 理解它和 `specify`、`tasks` 的边界
- 理解它产出的不仅是 `plan.md`

本文全部使用普通 Markdown 与 ASCII 图。

---

# 一、`plan` 是什么

一句话：

**`plan` 是把“要做什么”翻译成“准备怎么做”。**

它回答的问题是：

```text
这个目标，用什么路径实现最合理？
```

它不再停留在需求层，而是进入：

- 技术上下文
- 数据模型
- 接口契约
- 风险与约束
- 设计制品生成

---

# 二、先看图

## 1. 它在流程中的位置

```text
specify / clarify
       ↓
+------------------+
|      plan        |
|   把目标转成方案   |
+------------------+
       ↓
+------------------+
|      tasks       |
|   把方案拆成任务   |
+------------------+
```

核心意思：

**`plan` 负责“设计路径”，还不直接进入任务执行。**

---

## 2. 它的输入输出关系

```text
输入：
- spec.md
- constitution.md

输出：
- plan.md
- research.md
- data-model.md
- contracts/
- quickstart.md
```

所以 `plan` 不是单文件动作，而是一组设计制品的生成步骤。

---

# 三、形象表达

## 1. 它像什么

### 比喻 1：它像“施工前总方案会”

`specify` 说清楚了要造什么。

`plan` 才开始讨论：

- 用什么材料
- 怎么分区
- 风险怎么控
- 哪些接口要先定
- 哪些前置调研必须做

所以它像是：

**从需求进入设计的那场正式方案会。**

### 比喻 2：它像“把目标翻译成工程语言”

需求语言通常像：

- 希望更快
- 希望更稳定
- 支持某类用户场景

而 `plan` 要把它翻译成：

- 技术上下文
- 数据模型
- 接口约束
- 验证路径

这就是为什么 `plan` 是桥梁层。

### 比喻 3：它像“从地图进入路线规划”

`specify` 给你的是地图：
- 要去哪
- 为什么去
- 成功是什么

`plan` 给你的是路线：
- 走哪条路
- 经过哪些站点
- 哪些地方危险
- 到站后如何验证

---

# 四、硬核拆解

## 1. 输入是什么

主要输入有两类：

### 1. `spec.md`

提供：

- 用户故事
- 需求范围
- 成功标准
- 边界情况

### 2. `constitution.md`

提供：

- 项目原则
- 质量门禁
- 必须遵守的约束

这就是为什么 `plan` 模板里会有：

```text
Constitution Check
```

因为方案不是自由设计，而是要先过规则层。

---

## 2. 输出是什么

`plan` 的输出通常不止一个文件：

### A. `plan.md`

主方案文档，记录：

- 技术上下文
- 结构选择
- 约束
- 复杂度说明

### B. `research.md`

用于记录：

- 调研结论
- 技术选型理由
- 备选方案与取舍

### C. `data-model.md`

用于记录：

- 实体
- 字段
- 关系
- 状态迁移

### D. `contracts/`

用于记录：

- 接口契约
- 外部交互形式
- API / CLI / grammar / UI contract 等

### E. `quickstart.md`

用于记录：

- 基础验证方式
- 快速试运行路径
- 核心集成场景

---

## 3. 它的内部流程图

```text
[开始]
   ↓
读取 spec.md
   ↓
读取 constitution.md
   ↓
初始化 plan.md
   ↓
填写 Technical Context
   ↓
做 Constitution Check
   ↓
识别未知项 / 风险项
   ↓
生成 research.md
   ↓
提取实体与关系
   ↓
生成 data-model.md
   ↓
识别对外接口
   ↓
生成 contracts/
   ↓
生成 quickstart.md
   ↓
更新 agent 上下文
   ↓
再次做 Constitution Check
   ↓
[结束]
```

---

## 4. 每一步具体在做什么

### 第 1 步：读取 spec

这一步的目标不是重复需求，而是提取对设计有意义的信息，例如：

- 主要用户故事
- 成功标准
- 范围边界
- 关键对象

也就是说：

`plan` 不需要整个需求文案，它需要的是“能驱动设计的那部分”。

---

### 第 2 步：读取 constitution

这是 `plan` 很特别的一点。

因为在很多流程里，方案设计是自由发挥。

但在这里，方案要先问：

- 符不符合项目原则
- 会不会违反质量门禁
- 有没有引入不被允许的复杂度

所以 `constitution` 在这里像一个前置审查器。

---

### 第 3 步：填写 Technical Context

这是 `plan.md` 的核心区块之一。

它会尝试明确：

- 语言 / 版本
- 主要依赖
- 存储方式
- 测试方式
- 目标平台
- 项目类型
- 性能目标
- 约束
- 规模

这一步的意义在于：

**让方案从抽象目标变成有环境约束的可落地设计。**

---

### 第 4 步：做 Constitution Check

这一层问的是：

```text
这个设计是否违反了项目最高规则？
```

例如：

- 宪章要求简单优先，但你设计了过度复杂的结构
- 宪章要求测试优先，但方案完全没有验证路径
- 宪章要求可审计，但方案没考虑记录链路

这一步会把“方案质量”提前拉到设计阶段处理。

---

### 第 5 步：生成 research.md

这一步通常处理的是：

- 尚不明确的技术点
- 需要调研的方案选择
- 需要比较的替代路径

它的价值在于：

**把“为什么选这个方案”记录下来。**

否则后面很容易反复重走同一个判断过程。

---

### 第 6 步：生成 data-model.md

这一步负责把需求里的对象关系结构化。

它通常会明确：

- 有哪些核心实体
- 每个实体的关键字段
- 实体之间的关系
- 是否有状态迁移

这样后面的 `tasks` 和 `implement` 才不会在对象理解上各自发挥。

---

### 第 7 步：生成 contracts

如果系统有对外接口，就应该在这里把接口约束先定下来。

它不一定是 HTTP API，也可能是：

- CLI 输入输出格式
- UI interaction contract
- parser grammar
- 文件格式约定

这一层的核心是：

**先把边界接口定清楚，再拆任务。**

---

### 第 8 步：生成 quickstart

这一步是在回答：

```text
后面怎么快速验证这个方案和实现是否成立？
```

quickstart 的意义不是文档好看，而是给后面的实现和验证提供一条最短闭环路径。

---

### 第 9 步：更新 agent 上下文

这一步通常会调用：

- `update-agent-context.sh`
- 或 `update-agent-context.ps1`

作用是把当前方案中的技术上下文同步到 agent 文件里。

这意味着：

**`plan` 不只是写给人看，也是在更新 AI 的工作上下文。**

---

## 5. 它和前后步骤的边界

### 和 `specify` 的边界

`specify` 负责：

- 做什么
- 为什么做
- 成功是什么

`plan` 负责：

- 在什么技术上下文下做
- 用什么路径做
- 风险怎么控
- 验证路径是什么

### 和 `tasks` 的边界

`plan` 负责：

- 方案结构
- 设计制品
- 核心路径

`tasks` 负责：

- 把这些设计拆成阶段和任务

所以：

```text
plan = 设计路径
tasks = 执行拆解
```

---

## 6. 什么情况下最该重做 plan

这一步是非常允许重做的。

适合重做的场景包括：

- 方案不满意
- 风险识别不全
- 数据模型不够完整
- contracts 太粗
- quickstart 无法验证
- clarify 改动了关键需求

这是因为：

`plan` 是整个系统里最典型的“可迭代设计层”。

---

## 7. 常见误区

### 误区 1：把 plan 写成 task list

如果 `plan` 里已经变成：

- 先写这个
- 再做那个
- 再改这个文件

那你已经滑到 `tasks` 层了。

### 误区 2：没有记录方案取舍

如果只写“采用方案 X”，但没写为什么，
后面很容易反复重新争论。

### 误区 3：只写技术名词，不写约束和验证

方案不是堆技术名词，而是要解释：

- 为什么
- 风险
- 约束
- 怎么验证

### 误区 4：跳过 Constitution Check

如果不做这一步，方案就会脱离项目原则自由漂移。

---

## 8. 最后两句总结

### 形象版

**`plan` 像施工前的总方案会：它不是决定要不要建，而是决定怎么建才合理。**

### 硬核版

**`plan` 是方案层：它把需求规格翻译成技术上下文、结构设计、风险控制和验证路径，并生成后续拆解所需的设计制品。**

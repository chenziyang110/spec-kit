# Specify 详解：形象表达 + 硬核拆解

本文讲 Spec 方法论中的 `specify` 这一步。

目标：

- 用形象方式理解它在做什么
- 用结构化方式理解它的输入、输出和内部流程
- 理解它为什么是“目标层”，而不是“方案层”

本文全部使用普通 Markdown 与 ASCII 图，不依赖 Mermaid。

---

# 一、`specify` 是什么

一句话：

**`specify` 是把自然语言想法，压成一份可被后续流程消费的目标文档。**

它回答的是：

```text
我们到底要做什么，为什么做，边界在哪，成功是什么样？
```

它不回答：

```text
具体用什么技术做，结构怎么搭，任务怎么拆
```

那些属于后面的 `plan` 和 `tasks`。

核心意思：

**`specify` 负责“定义目标”，不是“设计实现”。**

# 三、形象表达

## 1. 它像什么

### 比喻 1：它像产品需求定稿会

不是讨论“怎么开发”，而是先讨论：

- 用户是谁
- 要解决什么问题
- 什么场景最重要
- 什么不做
- 成功怎么衡量

它像是把一段模糊的口头想法，压成一个能交给后面团队继续推进的正式需求说明。

### 比喻 2：它像打草稿，但必须是可执行草稿

普通草稿是随手写的。

`specify` 的草稿不是那种草稿，它必须能支撑后续几件事：

- 被 `clarify` 继续消歧
- 被 `plan` 转成方案
- 被 `tasks` 转成执行列表

所以它不是“想到什么写什么”，而是：

**把模糊输入整理成带结构的目标文档。**

### 比喻 3：它像施工前的“任务定义书”

假设你说：

> 做一个相册应用

这句话本身没法直接拿去做事，因为不清楚：

- 给谁用
- 核心价值是什么
- 什么功能算第一优先级
- 成功怎么判断

`specify` 的作用，就是把这句口头话变成：

```text
谁在什么场景下，要完成什么目标，系统必须提供什么能力，成功如何衡量
```

# 四、硬核拆解

## 1. 输入是什么

`specify` 的输入通常有三层：

### 1. 用户的自然语言描述

例如：

```text
Build an application that can help me organize my photos in separate photo albums...
```

### 2. 项目级原则

来自：

```text
.specify/memory/constitution.md
```

它会影响：

- 需求该写到什么程度
- 成功标准怎么写
- 什么模糊点不能放过

### 3. 规格模板

模板文件：

[spec-template.md](/F:/github/spec-kit/templates/spec-template.md)

它定义了 `spec.md` 应该有哪些标准区块，比如：

- 用户故事
- 功能需求
- 成功标准
- 边界情况
- 假设

---

## 2. 输出是什么

它的核心输出是：

```text
specs/<feature>/spec.md
```

附加输出通常还有：

```text
specs/<feature>/checklists/requirements.md
```

也就是说，它不只是写一份规格，还会顺手做一次“规格质量检查”。

---

## 3. 它的内部流程图

```text
[开始]
   ↓
接收用户自然语言描述
   ↓
生成 feature short name
   ↓
调用 create-new-feature 脚本
   ↓
创建功能分支 / 功能目录 / spec.md
   ↓
加载 spec-template
   ↓
提取关键概念
   ├─ actors
   ├─ actions
   ├─ data
   └─ constraints
   ↓
填充 spec 结构
   ├─ User Stories
   ├─ Functional Requirements
   ├─ Success Criteria
   ├─ Edge Cases
   └─ Assumptions
   ↓
做规格质量校验
   ↓
如果有关键歧义
   → 最多提出 3 个澄清问题
   → 用户回答
   → 回写 spec
   ↓
输出 spec 路径与下一步建议
   ↓
[结束]
```

---

## 4. 每一步在做什么

### 第 1 步：接收自然语言描述

这里的输入通常就是你在命令后面写的那段话。

例如：

```text
/speckit.specify 做一个帮助学生自学的课程平台
```

这一步的原始输入很可能是模糊的、混杂的、带跳跃的。

所以后面真正做的不是“照抄”，而是“结构化提取”。

---

### 第 2 步：生成 feature short name

它会先从描述里提取一个短名称，用于：

- 分支名
- 功能目录名

例如：

```text
photo-albums
student-learning-platform
family-budget-dashboard
```

这一步看起来像小事，但实际上是在给这个功能建立一个稳定标识。

---

### 第 3 步：创建功能分支和 spec 初始文件

它会调用：

- [create-new-feature.sh](/F:/github/spec-kit/scripts/bash/create-new-feature.sh)
- 或 [create-new-feature.ps1](/F:/github/spec-kit/scripts/powershell/create-new-feature.ps1)

这一步通常会做：

1. 创建 feature branch
2. 创建 `specs/<feature>/`
3. 初始化 `spec.md`

所以 `specify` 不只是“写文档”，它会顺带把 feature workspace 建起来。

---

### 第 4 步：提取关键概念

这一步会从原始描述中提取：

- actors：谁在用
- actions：要做什么
- data：涉及什么对象或数据
- constraints：有没有明显边界和限制

这一步本质上是在把原始描述转成一个“目标语义模型”。

---

### 第 5 步：填充 spec 的核心区块

#### A. 用户故事

把原始需求按用户价值切成若干独立旅程。

不是简单列功能点，而是组织成：

```text
谁
在什么场景下
要完成什么事
为什么重要
```

#### B. 功能需求

写成可测试、可引用的要求，通常带编号，例如：

```text
FR-001
FR-002
...
```

#### C. 成功标准

这里要求的是：

**可衡量、与技术无关、面向结果。**

例如：

- 用户能在 3 分钟内完成一次创建流程
- 90% 用户第一次即可完成核心任务

而不是：

- API 延迟 < 200ms
- Redis 命中率 > 80%

后者太偏实现层。

#### D. 边界情况

这里会开始捕捉：

- 空状态
- 错误情况
- 极端输入
- 异常路径

#### E. 假设

如果原始输入没说清，但又必须继续，它会把“合理默认值”写成假设。

这很关键，因为：

**默认值如果不显式写出来，后面就会变成隐性分歧。**

---

### 第 6 步：控制歧义

`specify` 不会无限提问。

它的设计是：

- 尽量做合理推断
- 只在关键处保留 `[NEEDS CLARIFICATION]`
- 最多保留少量关键澄清点

这意味着它不是“先把所有问题问完再动”，而是：

**尽量靠结构化推断推进，但把真正高影响的歧义显式标出来。**

---

### 第 7 步：质量校验

这是 `specify` 最重要的机制之一。

它不是写完就算，而是会检查：

- 是否混入实现细节
- 是否写得太模糊
- 是否所有必填章节都完成
- 成功标准是否可衡量
- 是否还残留关键 `[NEEDS CLARIFICATION]`

所以它不是单纯的“文档生成”，而是：

**目标文档生成 + 质量门禁。**

---

### 第 8 步：必要时向用户追问

如果还有关键歧义，它会提出有限数量的问题。

这些问题通常不是随便问，而是问那些会显著影响：

- 范围
- 安全/隐私
- 用户体验
- 成功标准

然后把答案回写到 `spec.md`。

---

## 5. 它不做什么

理解 `specify`，最重要的是知道它不负责什么。

它不负责：

- 技术选型
- 架构设计
- 数据库方案
- 接口设计细节
- 任务拆解
- 实现路径

这些都留给后面的：

- `clarify`
- `plan`
- `tasks`

所以如果在 `specify` 阶段写太多“怎么实现”，就会污染目标层。

---

## 6. 好的 specify 产物长什么样

好的 `spec.md` 通常具备这些特征：

### 特征 1：面向用户结果，而不是实现手段

### 特征 2：边界清晰

### 特征 3：成功标准可衡量

### 特征 4：用户故事能独立成立

### 特征 5：后面的人拿到它，知道要继续问什么、设计什么、拆什么

也就是说，一份好的 `spec` 不是“写得长”，而是“足以支撑下游”。

---

## 7. 常见误区

### 误区 1：把 `specify` 写成 plan

例如：

- 使用 React + FastAPI
- 用 PostgreSQL 做存储
- 用 Redis 做缓存

这些都不是 `specify` 层该回答的。

### 误区 2：只有功能点，没有用户故事

例如：

- 支持搜索
- 支持收藏
- 支持通知

这种写法太像功能列表，后面很难按用户价值拆解。

### 误区 3：成功标准写成技术指标

成功标准应该偏结果，不要一上来写成实现参数。

### 误区 4：不写假设

如果你靠默认值推进，却不把默认值写出来，后面就很容易出现“原来你是这么理解的”。

---

## 8. 最后两句总结

### 形象版

**`specify` 像把一句模糊想法，压成可以交给下游继续推进的正式任务定义书。**

### 硬核版

**`specify` 是目标层：它负责定义“做什么、为什么做、边界在哪、成功是什么”，但不负责定义“怎么做”。**

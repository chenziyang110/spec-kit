# Implement 详解：形象表达 + 硬核拆解

本文讲 Spec 方法论中的 `implement` 这一步。

目标：

- 理解它为什么是“执行层”
- 理解它不是自由发挥，而是沿着任务编排推进
- 理解它为什么会检查 checklist 和基础工程卫生

本文全部使用普通 Markdown 与 ASCII 图。

---

# 一、`implement` 是什么

一句话：

**`implement` 是把前面已经明确的任务计划，真正变成交付结果。**

它回答的问题是：

```text
现在按什么顺序动手，做到哪里算完成？
```

它不是重新定义需求，也不是重新设计方案。

它的重点是：

- 执行
- 回写状态
- 校验完成

---

# 二、先看图

## 1. 它在流程中的位置

```text
constitution
   ↓
specify
   ↓
clarify
   ↓
plan
   ↓
tasks
   ↓
analyze
   ↓
+------------------+
|    implement     |
|   按任务真正落地   |
+------------------+
```

核心意思：

**`implement` 是整个链条里最“重”的一层，但它不是最自由的一层。**

---

## 2. 它的前后关系

```text
tasks = 说明应该做什么
implement = 真正把它做出来
```

也就是说：

`implement` 不应该替代上游思考层。

如果上游有明显问题，正确动作通常是回退，而不是硬做。

---

# 三、形象表达

## 1. 它像什么

### 比喻 1：像正式开工

前面这些步骤都在做准备：

- 立规则
- 定目标
- 消歧义
- 做方案
- 拆任务

到了 `implement`，才是：

**真正开始施工。**

### 比喻 2：像按排程表施工，而不是现场 improvisation

它不是“想到哪做哪”，而是：

- 按阶段推进
- 按依赖推进
- 先做基础，再做故事
- 做完就回写状态

这意味着它更像工程执行，不像创意发散。

### 比喻 3：像总装线，不是实验台

如果你把 `implement` 当实验台，就会不停在这里临时改目标、改方案、改边界。

那就会导致：

- 边做边改
- 文档和实现漂移
- 任务状态失真

`implement` 的正确位置更像总装线：

**前面决定好了，这里按编排执行。**

---

# 四、硬核拆解

## 1. 输入是什么

`implement` 主要读取：

- `tasks.md`
- `plan.md`
- `data-model.md`（若有）
- `contracts/`（若有）
- `research.md`（若有）
- `quickstart.md`（若有）
- `checklists/`（若存在）

其中：

- `tasks` 决定动作顺序
- `plan` 决定方案边界
- 其他制品提供设计细节
- `checklists` 充当执行前的质量门禁

---

## 2. 输出是什么

`implement` 的输出不是单一文档，而是：

- 实际代码 / 文件变更
- 回写后的 `tasks.md`（完成项变成 `[X]`）
- 完成状态报告
- 如有需要，后置 hooks 的执行结果

所以它是真正把“文档世界”变成“结果世界”的那一步。

---

## 3. 它的内部流程图

```text
[开始]
   ↓
执行 before_implement hooks
   ↓
检查前置条件
   ↓
扫描 checklists/
   ↓
如果有未完成项
   → 询问是否继续
   ↓
读取 tasks / plan / 相关设计制品
   ↓
检查项目基础卫生
   ├─ ignore 文件
   ├─ 环境结构
   └─ 必要基础配置
   ↓
按阶段执行任务
   ├─ Setup
   ├─ Foundational
   ├─ User Story phases
   └─ Polish
   ↓
每完成一项任务就回写 [X]
   ↓
完成校验
   ↓
执行 after_implement hooks
   ↓
[结束]
```

---

## 4. 它为什么先检查 checklist

这是它很特别的一点。

它不是上来就开做，而是先看：

```text
前面的质量门禁有没有完成？
```

如果存在：

- `ux.md`
- `security.md`
- `requirements.md`

这些 checklist 还没完成，它会停下来确认是否要继续。

原因很简单：

**如果上游门禁都没过，就直接开工，返工概率会更高。**

---

## 5. 它为什么会检查 ignore 文件

`implement` 不只是写业务代码，它还会顺手做基础工程卫生检查。

例如检查：

- `.gitignore`
- `.dockerignore`
- `.eslintignore`
- `.prettierignore`
- `.terraformignore`

这背后的逻辑是：

**真正实现一个功能，不只是写逻辑，还包括把工程环境维持在可工作状态。**

所以 `implement` 不是纯编码命令，而是执行阶段的总入口。

---

## 6. 它怎么按阶段执行

典型顺序是：

### Phase 1：Setup

例如：

- 初始化目录
- 补基础配置
- 处理通用环境

### Phase 2：Foundational

例如：

- 基础 schema
- 认证骨架
- 通用中间件
- 通用日志 / 错误处理

### Phase 3+：按用户故事推进

每个故事作为一个独立增量来实现。

### Final Phase：Polish

例如：

- 文档
- 统一验证
- 性能 / 安全补强

这套顺序不是随便定的，而是为了保证：

**先打地基，再起价值切片，最后统一打磨。**

---

## 7. 它怎么处理任务状态

这是 `implement` 的硬规则之一：

**完成任务后，要把 `tasks.md` 中对应项标记为 `[X]`。**

这意味着：

- `tasks.md` 不是静态文档
- 它是执行中的状态账本

所以 `implement` 既在做事，也在同步进度。

---

## 8. 它和前面步骤的边界

### 不该在 implement 里做的事

不应该在这里再去：

- 重写目标
- 发明新需求
- 重新做大方案
- 重新发明任务结构

如果发现这些事情出了问题，通常应当回退：

- 目标错 -> 回 `specify`
- 歧义多 -> 回 `clarify`
- 方案差 -> 回 `plan`
- 拆解错 -> 回 `tasks`

### 它真正该做的事

- 按既定结构落地
- 把设计变成实现
- 持续回写状态
- 做完成校验

---

## 9. 什么情况下不应该继续 implement

如果你已经发现以下问题，就不适合继续硬推：

- `spec` 还很模糊
- `plan` 明显不稳
- `tasks` 排序明显不合理
- `analyze` 刚刚发现高优先级冲突

这时候最合理的动作通常是：

**回上游修，不是在实现层硬修。**

---

## 10. 常见误区

### 误区 1：把 implement 当作万能阶段

很多人会把所有没想清楚的东西都拖到实现时再决定。

这会让实现层承担本不属于它的认知负担。

### 误区 2：做完代码但不回写任务状态

那 `tasks.md` 就会失真，后面整个流程都会乱。

### 误区 3：不做完成校验

如果没有完成校验，就很容易在“自以为完成”的状态下结束。

---

## 11. 最后两句总结

### 形象版

**`implement` 像正式开工：前面图纸和排程已经准备好，现在按计划真正把东西做出来。**

### 硬核版

**`implement` 是执行层：它读取任务编排与设计制品，经过门禁检查后按阶段推进实现，并持续回写任务完成状态与最终校验结果。**

# Tasks 详解：形象表达 + 硬核拆解

本文讲 Spec 方法论中的 `tasks` 这一步。

目标：

- 理解它为什么是“拆解层”
- 理解它和 `plan`、`implement` 的区别
- 理解为什么任务必须按用户故事组织

本文全部使用普通 Markdown 与 ASCII 图。

---

# 一、`tasks` 是什么

一句话：

**`tasks` 是把方案文档翻译成可执行工作单元。**

它回答的问题是：

```text
现在具体先做什么、后做什么、哪些能并行、每一项对应什么目标？
```

它不再讨论“方案对不对”，而是开始讨论：

- 怎么拆
- 怎么排顺序
- 怎么组织执行

---

# 二、先看图

## 1. 它在流程中的位置

```text
plan
  ↓
+------------------+
|      tasks       |
|   方案拆成任务    |
+------------------+
  ↓
implement
```

核心意思：

**`tasks` 负责把“设计语言”变成“执行语言”。**

---

## 2. 它前后的转换关系

```text
spec = 目标语言
plan = 设计语言
tasks = 执行语言
```

所以 `tasks` 其实是一个翻译层：

把：

- 用户故事
- 技术方案
- 数据模型
- 接口契约

翻成：

- 任务编号
- 文件路径
- 依赖关系
- 并行关系
- 阶段顺序

---

# 三、形象表达

## 1. 它像什么

### 比喻 1：它像施工排程表

`plan` 说：

- 用什么方案
- 哪些结构存在
- 哪些约束要满足

但到真正执行前，还需要一层：

- 先挖地基还是先拉电
- 哪些工种可以同时进场
- 哪些必须等上一步完成

这就是 `tasks`。

### 比喻 2：它像“把设计图变成施工工单”

设计图不能直接施工。

因为它缺少：

- 具体动作
- 明确顺序
- 文件落点
- 并行边界

`tasks` 的作用，就是把“概念方案”压成“可执行工单”。

### 比喻 3：它像任务编排器，而不是待办清单

很多人把任务列表理解成 TODO。

但这里不是简单列事项，而是要形成：

```text
有顺序
有阶段
有依赖
有并行
有故事映射
```

所以它更像编排器，不只是列表。

---

# 四、硬核拆解

## 1. 输入是什么

`tasks` 的输入通常来自这些设计制品：

### 必需输入

- `plan.md`
- `spec.md`

### 可选输入

- `data-model.md`
- `contracts/`
- `research.md`
- `quickstart.md`

这些输入共同决定：

- 要做哪些事
- 哪些事重要
- 哪些事有依赖
- 哪些故事可独立交付

---

## 2. 输出是什么

它的核心输出是：

```text
specs/<feature>/tasks.md
```

这份文件不是随便写的任务清单，而是有严格格式要求的执行文档。

任务通常必须包含：

- checkbox
- task id
- 故事标签
- 是否可并行
- 文件路径

---

## 3. 它的内部流程图

```text
[开始]
   ↓
读取 plan.md
   ↓
读取 spec.md
   ↓
读取可选设计制品
   ├─ data-model.md
   ├─ contracts/
   ├─ research.md
   └─ quickstart.md
   ↓
提取用户故事
   ↓
识别共享基础设施
   ↓
识别阻塞前置条件
   ↓
按用户故事映射模型 / 服务 / 接口 / 测试
   ↓
建立依赖顺序
   ↓
标注可并行任务
   ↓
生成 tasks.md
   ↓
校验格式、覆盖与可执行性
   ↓
[结束]
```

---

## 4. 它的关键结构

典型的 `tasks.md` 会分成几类阶段：

### Phase 1：Setup

项目初始化、基础目录、工具配置等。

### Phase 2：Foundational

阻塞性前置条件。
这些通常必须在所有用户故事之前完成。

### Phase 3+：每个用户故事一个阶段

按优先级组织，例如：

- 用户故事 1（P1）
- 用户故事 2（P2）
- 用户故事 3（P3）

### Final Phase：Polish / Cross-Cutting

收尾、文档、性能、安全、统一验证等。

这个结构背后的思想是：

**按“可交付增量”组织，而不是按技术类型平铺。**

---

## 5. 为什么要按用户故事组织任务

这是 `tasks` 最核心的设计思想。

因为如果你按技术模块拆：

- 先写所有 model
- 再写所有 service
- 再写所有 UI

就会出现一个问题：

**你做了很多工作，但没有形成任何一个完整可交付的用户价值切片。**

按用户故事拆的好处是：

- 每个故事可以独立实现
- 每个故事可以独立测试
- 每个故事可以独立演示
- 每个故事都能作为 MVP 增量

这是一种明显偏交付导向的拆法。

---

## 6. 它具体怎么拆任务

### 第 1 类：共享任务

例如：

- 项目初始化
- lint / format
- 基础目录

这些放在 Setup。

### 第 2 类：阻塞任务

例如：

- 认证框架
- 基础 schema
- 路由骨架
- 通用错误处理

这些放在 Foundational。

### 第 3 类：故事内任务

每个用户故事内再继续拆成：

- 测试（如果要求）
- 模型
- 服务
- 接口 / 页面 / 端点
- 集成

这一步实际上是在把 `plan` 中的设计元素映射回用户价值。

---

## 7. 依赖与并行是怎么处理的

`tasks` 的关键不是任务数量，而是编排关系。

它会区分：

### 串行任务

必须按顺序做，因为：

- 有依赖
- 影响同一文件
- 上一步没完成，下一步无意义

### 并行任务

可以同时做，因为：

- 文件不同
- 无强依赖
- 不会互相阻塞

所以任务里通常会出现 `[P]` 标记。

这意味着：

**`tasks` 不是列事情，而是在做任务调度。**

---

## 8. 它和前后步骤的边界

### 和 `plan` 的边界

`plan` 负责：

- 设计路径
- 技术上下文
- 数据结构
- 风险控制

`tasks` 负责：

- 把这些方案拆成执行单元

### 和 `implement` 的边界

`tasks` 负责：

- 说明该做什么
- 说明顺序和依赖

`implement` 负责：

- 真正去执行这些任务
- 完成后把任务标记为 `[X]`

所以：

```text
tasks = 编排
implement = 落地
```

---

## 9. 什么情况下最该重做 tasks

这一步非常适合重做。

需要重做的场景包括：

- `plan` 改了
- 依赖顺序不合理
- 粒度太粗或太细
- 用户故事映射不清
- 测试任务缺失
- 非功能任务遗漏

因为任务层本质上就是：

**把设计压缩成执行结构。**

只要设计变了，任务通常就该跟着变。

---

## 10. 常见误区

### 误区 1：把 tasks 写成 plan

如果任务写成：

- 采用某某架构
- 使用某某方案

那还是方案层，不是任务层。

任务应该是动作，而不是设计判断。

### 误区 2：只有技术模块，没有用户故事映射

这样会导致“做了很多，但没有可交付切片”。

### 误区 3：没有文件路径

没有落点的任务容易变得很虚。

### 误区 4：没有依赖顺序

没有顺序的任务清单只是愿望清单，不是执行系统。

---

## 11. 最后两句总结

### 形象版

**`tasks` 像施工排程表：不是告诉你房子怎么设计，而是告诉你今天、明天、后天分别该干什么。**

### 硬核版

**`tasks` 是拆解层：它把方案文档翻译成带编号、带依赖、可并行、可执行的工作单元，并按用户故事组织成交付增量。**
